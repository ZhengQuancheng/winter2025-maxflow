# 最大流算法 GPU 加速项目总结报告

## 一、项目概述

本项目实现了一套最大流(Max Flow)算法程序，包括图数据生成、CPU 基线算法、GPU 加速算法和结果校验四个模块，并在五款国产及国际主流 GPU 平台(包括 NVIDIA、天数智芯、沐曦、摩尔线程和海光)上进行性能评测。

| 模块 | 文件 | 功能 |
|------|------|------|
| 图生成器 | `gen_graph.cpp` | 生成随机有向图，输出 CSR 二进制格式 |
| CPU 求解器 | `cpu_maxflow.cpp` | Dinic 算法 + Edmonds-Karp 算法 |
| GPU 求解器 | `gpu_maxflow.cu` | 并行 Push-Relabel 算法，支持 5 种 GPU 平台 |
| 结果校验器 | `check_results.cpp` | 检查 GPU 加速算法与 CPU 基线算法的结果误差 |   

测试规模：节点数 N ∈ {1000, 5000, 10000, 50000, 100000, 500000}，每组 100 次查询。

---

## 二、图数据生成（gen_graph.cpp）

### 2.1 实现思路

图生成器的核心目标是产生**保证 s→t 可达**的随机有向图，同时控制内存占用。

**Hamiltonian 路径骨架**：生成器首先构造一条从节点 0 经过随机打乱的内部节点最终到达节点 N-1 的哈密顿路径，作为图的连通骨架。所有查询对 (s, t) 均从该路径上选取满足 `pos_s < pos_t` 的节点对，从而在算法层面保证可达性，无需运行时验证。

**随机额外边**：在骨架之上随机添加 `M - (N-1)` 条额外有向边，边容量均匀分布于 [1.0, 100.0]。

**平行边合并**：对 (u, v) 相同的平行边，将其容量累加合并为单条边，保证 CSR 格式的唯一性。

**内存预算控制**：总边数上限约 2600 万条(对应约 512 MB 内存)，超出时自动截断并警告。

### 2.2 CSR 二进制格式

输出文件格式（小端序）：

```
[int N] [int M] [int row_ptr[N+1]] [int col[M]] [float cap[M]]
```

该格式可被 CPU 和 GPU 求解器直接 mmap 或 fread 加载，无需格式转换。

## 三、CPU 基线算法（cpu_maxflow.cpp）

### 3.1 残量图构建

CPU 求解器在原始图基础上构建残量图（Residual Graph）。对每条原始边 u→v（容量 c），在残量图中同时插入：
- 正向边 u→v，初始残量 = c
- 反向边 v→u，初始残量 = 0

通过 `rev[]` 数组记录每条边与其反向边的索引对应关系，支持 O(1) 反向边查找。残量图边数为原始图的 2 倍。

### 3.2 Dinic 算法

Dinic 算法分两阶段交替执行：

**BFS 构建层次图**：从源点 s 出发，按残量图中的残余容量做 BFS，为每个可达节点赋予层次编号 `level[v]`。若汇点 t 不可达则算法终止。

**迭代 DFS 寻找阻塞流**：在层次图约束下（仅走满足 `level[to] == level[v] + 1` 的边），用迭代式 DFS 栈反复寻找增广路并推送流量，直到层次图中不存在增广路为止。

**当前弧优化**：每个节点维护一个当前弧指针 `it[v]`，DFS 从当前弧开始扫描，跳过已饱和的边，避免重复遍历，将单次阻塞流的时间复杂度从 O(VE) 降至 O(VE)（常数更小）。

**回退时标记死节点**：DFS 回退时将无法继续推流的节点 `level[v]` 置为 -1，后续路径不再经过该节点，进一步减少无效搜索。

### 3.3 Edmonds-Karp 算法

Edmonds-Karp 是 Ford-Fulkerson 的 BFS 变体，每次用 BFS 寻找**跳数最短**的增广路，沿路径推送瓶颈流量后更新残量图，重复直到汇点不可达。

相比 Dinic，Edmonds-Karp 每次只找一条增广路，迭代次数更多，时间复杂度为 O(VE²)，在大图上性能显著落后。

## 四、GPU 加速算法（gpu_maxflow.cu）

### 4.1 多平台抽象层

GPU 求解器通过宏实现对五种 GPU 平台的统一支持，无需维护多份代码：

| 平台 | 运行时头文件 | 宏前缀 |
|------|------------|--------|
| NVIDIA (CUDA) | `cuda_runtime.h` | `cuda` |
| 天数智芯 Iluvatar | `cuda_runtime.h` | `cuda`（兼容层） |
| 摩尔线程 Moore | `musa_runtime.h` | `musa` |
| 沐曦 MetaX | `mc_runtime.h` | `mc` |
| 海光 Hygon | `hip/hip_runtime.h` | `hip` |

所有平台 API 统一映射为 `GPU*` 前缀宏（如 `GPUMalloc`、`GPUMemcpy`），上层算法代码完全平台无关。

### 4.2 算法选择：Push-Relabel

GPU 求解器采用 **Push-Relabel（预流推进）** 算法，而非 Dinic。原因在于：

- Push-Relabel 的每步操作（push/relabel）是**局部的、节点级别的**，天然适合 GPU 的 SIMT 并行模型——每个线程独立处理一个节点，无需全局同步。
- Dinic 的 BFS + DFS 路径搜索存在串行依赖，难以高效并行化。

### 4.3 算法流程

**预流初始化（kernel_saturate_source）**：
将源点 s 的所有出边全部饱和推送，使邻居节点获得初始盈余（excess），源点高度初始化为 V，其余节点高度为 0。

**push-relabel（kernel_discharge）**：
每个线程负责一个节点 u，执行 fused discharge 操作：
1. 原子地取走 `excess[u]`（`atomicExch`），避免多线程竞争同一节点
2. **第一轮 push**：扫描 u 的所有出边，对满足 `height[u] == height[v] + 1` 的 admissible edge 推送流量
3. **relabel**：若仍有剩余盈余，将 `height[u]` 提升至所有有残量出边邻居高度的最小值 + 1
4. **第二轮 push**：relabel 后立即再次尝试推送，避免等待下一轮迭代

两轮 push-relabel 融合在单次 kernel 调用中完成，减少 kernel 启动开销。

**全局重标号（global_relabel）**：
每 32 轮 discharge 执行一次两阶段 BFS 全局重标号：
- **Phase 1（反向 BFS from t）**：沿残量图反向边从汇点 t 出发，为所有能到达 t 的节点计算精确高度
- **Phase 2（正向 BFS from s）**：从源点 s 出发，为"可从 s 到达但无法到达 t"的节点赋予 V + 距离的高度（标记为不可用）

全局重标号大幅减少无效的 relabel 操作，是 Push-Relabel 最重要的加速启发式。

**收敛判断**：
每 8 轮 discharge 检查一次：若 `flag == 0`（无节点做过有效工作）且所有活跃节点的最大盈余 ≤ EPS，则算法收敛。

### 4.4 关键优化

**算子融合**：
两轮 push-relabel 融合在单次 kernel 调用中完成，减少 kernel 启动开销。

**Pinned Memory**：
将 CPU-GPU 之间频繁传输的小型控制数据（flag、fsz、max_bits、edge_updates）分配在 Pinned Memory 中，避免操作系统换页，实现 DMA 直传，降低同步延迟。

**双缓冲 BFS 前沿队列**：
BFS 使用两个交替使用的前沿队列缓冲区（`fa`、`fb`），当前层读 `cur`，下一层写 `nxt`，每轮交换指针，无需额外内存拷贝。

**Warp 级归约**：
计算全局最大盈余时，使用 `__shfl_down_sync` 在 warp 内做树形归约，每个 warp 只需一次 `atomicMax` 写全局内存，大幅减少原子操作竞争。Hygon 平台使用 `__shfl_down` 替代（HIP 兼容）。

| 图规模 | H800 | Hygon BW | Iluvatar BI100 | MetaX C500 | Moore S4000 |
|--------|------|----------|----------------|------------|-------------|
| N=1K | 1.08 M/s | 0.53 M/s | 0.46 M/s | 0.33 M/s | 0.22 M/s |
| N=10K | 1.47 M/s | 0.66 M/s | 0.59 M/s | 0.40 M/s | 0.28 M/s |
| N=50K | 1.52 M/s | 0.66 M/s | 0.47 M/s | 0.39 M/s | 0.18 M/s |
| N=100K | 1.42 M/s | 0.61 M/s | 0.34 M/s | 0.35 M/s | 0.11 M/s |

H800 吞吐量在 N=50K 时达到峰值（1.52 M updates/s），N=500K 时下降至 0.89 M/s，原因是大图的残量数组超出 L2 缓存容量，内存带宽成为瓶颈。

### 5.4 迭代次数分析

各平台在相同图上的平均迭代次数高度一致（误差 < 2%），说明算法收敛行为与硬件无关，迭代次数随图规模对数增长：

| N | 平均迭代次数 |
|---|------------|
| 1,000 | ~46 次 |
| 10,000 | ~54 次 |
| 100,000 | ~61 次 |
| 500,000 | ~69 次 |

这一增长趋势符合 Push-Relabel 配合全局重标号的理论预期（迭代次数约为 O(V) 量级，实际远小于理论上界）。

### 5.5 平台横向对比

按 N=100K 综合性能排名：

```
NVIDIA H800 > Hygon BW > Iluvatar BI100 ≈ MetaX C500 > Moore S4000
```

- **H800**：算力和内存带宽最强，全规模领先，N=500K 时仍保持 0.89 M/s 吞吐量
- **Hygon BW**：基于 HIP/ROCm 生态，性能稳定，大图表现优于 Iluvatar 和 MetaX
- **Iluvatar BI100 / MetaX C500**：中等规模（N≤100K）性能相近，大图（N=500K）时 MetaX 略优
- **Moore S4000**：在所有规模下性能最弱，N=500K 时平均查询时间高达 84 ms，为 H800 的 24.6 倍

### 5.6 首次查询延迟

各平台首次查询时间普遍高于后续查询（尤其 Hygon BW 约 15-23 ms），H800 首次查询延迟较低（1.5-4 ms）。

## 六、未来可继续提升的方向

### 6.1 算法层面

**自适应全局重标号频率**：
当前固定每 32 轮执行一次全局重标号（GR_FREQ=32）。可根据每轮 discharge 的有效推流比例动态调整频率——推流效率低时更频繁地重标号，推流效率高时减少重标号开销。

**图预处理（死端剪枝）**：
在求解前删除无法参与 s→t 流路径的节点和边（如出度为 0 的非汇点、入度为 0 的非源点），减少 discharge kernel 的无效线程。

### 6.2 GPU 实现层面

**Warp 级协作 discharge**：
当前每个线程独立处理一个节点。对于出度极高的节点，其出边扫描成为单线程瓶颈。可将一个 warp（32 线程）协作处理一个高度节点，用 warp-level 并行扫描出边，提升高度节点的处理吞吐量。

**共享内存缓存邻接表**：
对于出度较小的节点，将其邻接表（col、cap 片段）预加载到 shared memory，减少全局内存访问延迟，尤其对 L2 缓存命中率低的大图效果显著。

## 七、总结

本项目构建了一套从图生成、CPU 基线到 GPU 加速的完整最大流计算框架，核心贡献如下：

1. **算法适配**：选用 Push-Relabel 替代 Dinic，充分利用 GPU SIMT 并行性，在 N=500K 的大图上相对 CPU Dinic 实现最高 **241 倍**加速（NVIDIA H800）。

2. **平台适配**：通过宏实现单套代码支持 NVIDIA CUDA、天数智芯、摩尔线程、沐曦、海光五大平台，维护成本低。

3. **性能优化**：融合了全局重标号、Pinned Memory、两轮 push-relabel 融合、双缓冲队列、Warp 归约等多项优化，在各平台上均取得稳定的性能表现。
